🟢 Setup & Basics

Initialize repositories, clone projects, and configure your Git environment.

git init Basic

Initialize a new Git repository in the current directory

Syntax git init [directory]

Creates a new .git subdirectory containing all necessary repository metadata. This is the first command you run in a new project. It does not affect existing files.

  • Creates the .git/ folder with objects, refs, HEAD, and config
  • Sets the default branch (usually main or master)
  • No commits are created – you start with a clean slate

Click Play to see how git init sets up a repository
💡 Tip
Use git init --bare to create a bare repository (no working directory) – typically used for remote/server repos.

git clone Basic

Clone a repository into a new directory

Syntax git clone <url> [directory]

Creates a copy of an existing repository. It automatically sets up a remote called origin pointing to the cloned URL, downloads all branches and history, and checks out the default branch.

FlagDescription
--depth <n>Create a shallow clone with only the last n commits
--branch <name>Clone a specific branch instead of the default
--single-branchClone only the history of a single branch
--recurse-submodulesInitialize and clone submodules as well

git config Basic

Get and set repository or global options

Syntax git config [--global | --local | --system] <key> <value>

Configure Git settings at different levels. The most common use is setting your user name and email.

LevelScopeFile Location
--systemAll users on the machine/etc/gitconfig
--globalCurrent user, all repos~/.gitconfig
--localCurrent repository only.git/config
Examples git config --global user.name "Your Name"
git config --global user.email "you@example.com"
git config --list

📸 Snapshotting

Stage changes, create commits, inspect differences, and view history.

git add Basic

Add file contents to the staging area (index)

Syntax git add <pathspec>...

Moves changes from the Working Directory to the Staging Area (also called the Index). This is the step before committing — it lets you prepare exactly which changes go into the next commit.

FlagDescription
git add .Stage all changes in current directory and below
git add -AStage all changes in the entire working tree
git add -pInteractively select hunks to stage
git add *.jsStage all .js files

git commit Basic

Record changes to the repository

Syntax git commit -m "message"

Takes the staged snapshot and permanently stores it in the repository history. Each commit has a unique SHA-1 hash, author information, timestamp, and a pointer to its parent commit(s).

FlagDescription
-m "msg"Commit with an inline message
-aAutomatically stage all tracked modified files before committing
--amendReplace the last commit with a new one (edit message or add changes)
--no-editAmend without changing the commit message

git status Basic

Show the working tree status

Syntax git status [-s | --short]

Displays the state of the working directory and staging area. It shows which files are staged, unstaged, and untracked.

  • Green = staged (ready to commit)
  • Red = modified but not staged
  • Untracked files are listed separately
💡 Tip
Use git status -s for a compact two-column output. First column = staging area, second = working tree.

git diff Basic

Show changes between commits, branches, or working tree

Syntax git diff [--staged] [<commit>] [-- <path>]
  • git diff — changes in working dir vs staging area
  • git diff --staged — changes in staging area vs last commit
  • git diff branch1..branch2 — differences between two branches
  • git diff HEAD~3..HEAD — changes in the last 3 commits

git stash Advanced

Temporarily shelve changes for later use

Syntax git stash [push | pop | list | drop | apply]

Saves your uncommitted changes (both staged and unstaged) to a stash stack and reverts the working directory to HEAD. Useful when you need to switch branches but aren't ready to commit.

CommandDescription
git stashStash current changes
git stash popApply most recent stash and remove it from stack
git stash applyApply most recent stash but keep it in stack
git stash listList all stashed entries
git stash dropRemove the most recent stash
git stash -uInclude untracked files in the stash
🎬 Animation: git stash / pop
Click Play to see changes get stashed and popped

git log Basic

Show commit history

Syntax git log [--oneline] [--graph] [--all]

Displays the commit history starting from the current HEAD. Combine flags for different views:

  • git log --oneline — one commit per line
  • git log --graph --oneline --all — ASCII art tree of all branches
  • git log --author="name" — filter by author
  • git log -n 5 — show only last 5 commits
  • git log --since="2024-01-01" — commits after a date

🌿 Branching & Merging

Create branches, switch contexts, merge changes, and rewrite history.

git branch Branch

List, create, or delete branches

Syntax git branch [name] [-d | -D] [-a] [-r]

Branches are lightweight pointers to commits. Creating a branch is almost instant — it just creates a new pointer.

CommandDescription
git branchList local branches (* marks current)
git branch <name>Create a new branch at current HEAD
git branch -d <name>Delete branch (safe – only if merged)
git branch -D <name>Force delete branch
git branch -aList all branches (local + remote)
git branch -m <old> <new>Rename a branch
🎬 Animation: git branch
Click Play to see a new branch being created

git checkout Branch

Switch branches or restore working tree files

Syntax git checkout <branch>
git checkout -b <new-branch>

Switches HEAD to a different branch and updates the working directory. With -b, it creates a new branch and switches to it in one command.

  • git checkout main — switch to main branch
  • git checkout -b feature — create & switch to "feature" branch
  • git checkout -- file.txt — discard changes to a file (legacy; use git restore)
💡 Modern Alternative
Since Git 2.23, prefer git switch for branch switching and git restore for file restoring. checkout was overloaded with too many responsibilities.

git switch Branch

Switch branches (modern replacement for checkout)

Syntax git switch <branch>
git switch -c <new-branch>

A focused command that only switches branches (introduced in Git 2.23). Clearer intent than git checkout.

  • git switch main — switch to main
  • git switch -c feature — create & switch
  • git switch - — switch to previous branch

git merge Branch

Combine branch histories together

Syntax git merge <branch> [--no-ff] [--squash]

Integrates changes from one branch into the current branch. There are two main types:

  • Fast-forward merge — if the current branch hasn't diverged, Git just moves the pointer forward. No merge commit created.
  • Three-way merge — if branches have diverged, Git creates a new merge commit with two parents.
FlagDescription
--no-ffForce a merge commit even if fast-forward is possible
--squashCombine all changes into one commit (no merge commit)
--abortAbort a merge in progress (during conflicts)
🎬 Animation: git merge (3-way)
Click Play to see two branches merge together
⚠️ Merge Conflicts
If the same lines were changed in both branches, Git will pause and mark conflicts. Edit the files, then git add and git commit to complete the merge.

git rebase Advanced

Reapply commits on top of another base

Syntax git rebase <base-branch>
git rebase -i <commit>

Takes the commits from your current branch and replays them on top of the target branch, creating a linear history without merge commits.

  • Standard rebase — moves your branch to start from the tip of the base branch
  • Interactive rebase (-i) — lets you squash, reword, reorder, or drop commits
🎬 Animation: git rebase
Click Play to see commits rebased onto main
⚠️ Golden Rule
Never rebase public/shared branches. Rebase rewrites commit hashes, which causes problems for anyone who has already pulled the original commits.

git cherry-pick Advanced

Apply specific commits from another branch

Syntax git cherry-pick <commit-hash>

Copies a single commit (or multiple) from one branch and applies it on top of your current branch. Creates new commits with different hashes but the same changes.

🌐 Remote Operations

Collaborate with others using remote repositories.

git remote Remote

Manage remote repository connections

Syntax git remote [add | remove | rename | -v] <name> <url>

Remotes are bookmarks to other repositories. origin is the conventional name for the primary remote.

CommandDescription
git remote -vList remotes with URLs
git remote add origin <url>Add a remote named origin
git remote remove <name>Remove a remote
git remote rename <old> <new>Rename a remote

git fetch Remote

Download objects and refs from a remote

Syntax git fetch [remote] [--all] [--prune]

Downloads commits, files, and refs from a remote repository into your local repo. It does NOT merge anything — your working directory stays untouched. Safe to run anytime.

🎬 Animation: git fetch
Click Play to see remote commits downloaded locally

git pull Remote

Fetch and integrate remote changes

Syntax git pull [remote] [branch] [--rebase]

git pull = git fetch + git merge. It downloads remote changes and immediately merges them into your current branch.

  • git pull — fetch + merge from tracking branch
  • git pull --rebase — fetch + rebase instead of merge (cleaner history)
  • git pull origin main — pull specifically from origin/main

Click Play to see fetch + merge in action

git push Remote

Upload local commits to a remote repository

Syntax git push [remote] [branch] [-u | --force]

Uploads your local commits to the remote repository, updating the remote branch.

CommandDescription
git pushPush current branch to its upstream
git push -u origin mainPush and set upstream tracking
git push --forceForce push (overwrites remote – dangerous!)
git push --force-with-leaseSafer force push (fails if remote has new commits)
git push --tagsPush all tags to remote

⏪ Undo & History

Undo mistakes, reset state, and manage versions.

git reset Undo

Reset current HEAD to a specified state

Syntax git reset [--soft | --mixed | --hard] <commit>

Moves the current branch pointer back to a previous commit. The three modes control what happens to your changes:

  • --soft — moves HEAD only. Staging area and working dir unchanged. Changes become "staged".
  • --mixed (default) — moves HEAD and resets staging area. Working dir unchanged. Changes become "unstaged".
  • --hard — moves HEAD, resets staging area AND working directory. All changes are lost!
🎬 Animation: git reset modes
Click Play to compare --soft, --mixed, --hard
⚠️ Warning
git reset --hard permanently destroys uncommitted changes. Use with caution! If you've already pushed, prefer git revert instead.

git revert Undo

Create a new commit that undoes a previous commit

Syntax git revert <commit>

Instead of deleting history (like reset), revert creates a new commit that applies the inverse of the specified commit. This is safe for shared/public branches.

🎬 Animation: git revert
Click Play to see a revert commit being created

git restore Undo

Restore working tree files or unstage changes

Syntax git restore <file>
git restore --staged <file>
  • git restore file.txt — discard changes in working directory (revert to last commit)
  • git restore --staged file.txt — unstage a file (opposite of git add)
  • git restore --source=HEAD~2 file.txt — restore file from 2 commits ago

git tag Advanced

Create, list, and manage tags (version markers)

Syntax git tag <tagname> [-a] [-m "message"] [commit]

Tags mark specific commits as important — usually for releases (v1.0, v2.3.1, etc.).

  • Lightweight tag — just a name pointing to a commit: git tag v1.0
  • Annotated tag — stores metadata (author, date, message): git tag -a v1.0 -m "Release 1.0"
CommandDescription
git tagList all tags
git tag v1.0Create lightweight tag at HEAD
git tag -a v1.0 -m "msg"Create annotated tag
git tag -d v1.0Delete a tag
git push origin v1.0Push a specific tag to remote
